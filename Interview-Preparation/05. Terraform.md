# 📘 Terraform Full Notes (Easy Explanation)
---
## 🌟 What is Terraform?
**Terraform** is an open-source tool by **HashiCorp** used to define and provision infrastructure using **Infrastructure as Code (IaC)**.

> You write your infrastructure in simple text files using `.tf` or `.tf.json`, and Terraform automates creating, changing, or destroying resources like EC2, S3, VPC, etc.


## 🧩 Key Concepts (Modules)

### 🔢 1. VARIABLES
**Why?**  
Variables help you make your code flexible and reusable.

```hcl
variable "region" {
  default = "ap-south-1"
}

variable "instance_type" {
  default = "t2.micro"
}
```

**Use:**
```hcl
provider "aws" {
  region = var.region
}
```

---

### 🏗️ 2. RESOURCES
**Why?**  
Resources are the actual infrastructure components you want to create (like EC2, S3).

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0abcdef1234567890"
  instance_type = var.instance_type
}
```

---

### 🔍 3. DATA SOURCES
**Why?**  
To read or fetch information from existing infrastructure (you don’t create with it).

```hcl
data "aws_ami" "ubuntu" {
  most_recent = true
  owners      = ["099720109477"]

  filter {
    name   = "name"
    values = ["ubuntu/images/hvm-ssd/ubuntu-focal-*"]
  }
}

resource "aws_instance" "server" {
  ami           = data.aws_ami.ubuntu.id
  instance_type = "t2.micro"
}
```

---

### 🌐 4. PROVIDERS
**Why?**  
They define **which platform** (like AWS, Azure, GCP) Terraform should interact with.

```hcl
provider "aws" {
  region = var.region
}
```

---

### 📂 5. STATE FILE
**Why?**  
Terraform uses a **state file** (`terraform.tfstate`) to **track** what is created.

- It compares your code vs what exists.
- Helps with `plan`, `apply`, and `destroy`.
- Don’t modify it manually.

---

### ☁️ 6. REMOTE BACKEND
**Why?**  
Store your **state file** in a shared location (e.g., S3) so teams can collaborate.

```hcl
terraform {
  backend "s3" {
    bucket = "my-terraform-state"
    key    = "dev/terraform.tfstate"
    region = "ap-south-1"
  }
}
```

---

### 🔧 7. PROVISIONERS
**Why?**  
To run commands/scripts **after** a resource is created.

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0abcdef1234567890"
  instance_type = "t2.micro"

  provisioner "remote-exec" {
    inline = [
      "sudo apt update",
      "sudo apt install nginx -y"
    ]
  }

  connection {
    type        = "ssh"
    user        = "ubuntu"
    private_key = file("~/.ssh/id_rsa")
    host        = self.public_ip
  }
}
```

---

### 📦 8. MODULES
**Why?**  
Modules let you **reuse Terraform code** across different projects or environments.

**Structure:**
```
root/
├── main.tf
├── variables.tf
└── modules/
    └── ec2/
        ├── main.tf
        ├── variables.tf
        └── outputs.tf
```

**Use Module:**
```hcl
module "my_ec2" {
  source        = "./modules/ec2"
  instance_type = "t2.micro"
  ami_id        = "ami-0abcdef1234567890"
}
```

---

### 🧪 9. WORKSPACES
**Why?**  
To manage **multiple environments** like `dev`, `test`, `prod` using the same code.

**Commands:**
```bash
terraform workspace list
terraform workspace new dev
terraform workspace select dev
terraform apply
```

---

### 🔐 10. SECRETS & VAULT
**Why?**  
To avoid hardcoding secrets like access keys, use **Vault** for secure secret management.

```hcl
provider "vault" {
  address = "http://127.0.0.1:8200"
  token   = var.vault_token
}

data "vault_generic_secret" "aws_creds" {
  path = "secret/aws"
}

provider "aws" {
  access_key = data.vault_generic_secret.aws_creds.data["access_key"]
  secret_key = data.vault_generic_secret.aws_creds.data["secret_key"]
  region     = var.region
}
```

## 🔄 Common Terraform Commands

| Command              | Description                           |
|----------------------|---------------------------------------|
| `terraform init`     | Initialize working directory          |
| `terraform plan`     | Show what Terraform will do           |
| `terraform apply`    | Apply the configuration               |
| `terraform destroy`  | Delete all infrastructure             |
| `terraform fmt`      | Format code properly                  |
| `terraform validate` | Check config correctness              |

# 💼 Top 10 Terraform Interview Questions & Answers 

---

## ❓ 1. What is Terraform and why is it used?

**💬 Answer:**  
Terraform is an open-source **Infrastructure as Code (IaC)** tool developed by HashiCorp. It enables DevOps engineers to define and provision infrastructure using a declarative configuration language. It supports multi-cloud platforms and allows version control and repeatable deployments.



## 🔌 2. What are Terraform Providers?

**💬 Answer:**  
Terraform providers are plugins that allow Terraform to interact with external platforms like AWS, Azure, GCP, GitHub, etc. Each provider has its own set of resources and data sources.

```hcl
provider "aws" {
  region = "ap-south-1"
}
```



## 🗃️ 3. What is a Terraform State File and why is it important?

**💬 Answer:**  
The `terraform.tfstate` file stores the current state of infrastructure. Terraform uses it to compare the current state with the desired state in configuration files, and makes changes accordingly.

- Keep it secure (contains sensitive data)
- Can be stored remotely (e.g., S3) for team collaboration



## 🔁 4. Difference between `terraform plan` and `terraform apply`?

**💬 Answer:**  
- `terraform plan`: Shows what changes will be made (safe preview)
- `terraform apply`: Executes and applies the planned changes to infrastructure



## 📦 5. What are Modules in Terraform?

**💬 Answer:**  
Modules are reusable blocks of Terraform configuration. They help reduce code duplication and improve maintainability.

```hcl
module "vpc" {
  source = "./modules/vpc"
}
```



## 📥 6. What is the use of `terraform import`?

**💬 Answer:**  
`terraform import` is used to import existing infrastructure into Terraform’s management.

```bash
terraform import aws_instance.myec2 i-0abcd1234efgh5678
```



## 🔐 7. How do you manage secrets in Terraform?

**💬 Answer:**  
- Avoid hardcoding secrets in `.tf` files
- Use **Vault**, **SSM Parameter Store**, or **Secrets Manager**
- Mark variables as sensitive

```hcl
variable "db_password" {
  type      = string
  sensitive = true
}
```



## ☁️ 8. What is a Remote Backend?

**💬 Answer:**  
Remote backend stores the Terraform state file in a remote location like S3, Azure Blob, etc. It enables team collaboration and state locking (e.g., with DynamoDB).

```hcl
terraform {
  backend "s3" {
    bucket = "mybucket"
    key    = "state/terraform.tfstate"
    region = "ap-south-1"
  }
}
```


## 🧪 9. What are Terraform Workspaces?

**💬 Answer:**  
Workspaces allow managing multiple environments (dev, test, prod) using the same configuration.

```bash
terraform workspace new dev
terraform workspace select dev
terraform apply
```



## 📏 10. What are some Terraform best practices?

**💬 Answer:**  
- Use version control (Git)
- Separate resources using modules
- Use variables, locals, and outputs efficiently
- Use `terraform plan` before `apply`
- Store state remotely for team projects
- Use `.gitignore` to exclude `.tfstate` files
