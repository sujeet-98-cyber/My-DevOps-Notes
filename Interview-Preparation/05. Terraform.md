## ЁЯМЯ What is Terraform?
**Terraform** is an open-source tool by **HashiCorp** used to define and provision infrastructure using **Infrastructure as Code (IaC)**.

> You write your infrastructure in simple text files using `.tf` or `.tf.json`, and Terraform automates creating, changing, or destroying resources like EC2, S3, VPC, etc.


## ЁЯзй Key Concepts (Modules)

### ЁЯФв 1. VARIABLES
**Why?**  
Variables help you make your code flexible and reusable.

```hcl
variable "region" {
  default = "ap-south-1"
}

variable "instance_type" {
  default = "t2.micro"
}
```

**Use:**
```hcl
provider "aws" {
  region = var.region
}
```

---

### ЁЯПЧя╕П 2. RESOURCES
**Why?**  
Resources are the actual infrastructure components you want to create (like EC2, S3).

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0abcdef1234567890"
  instance_type = var.instance_type
}
```

---

### ЁЯФН 3. DATA SOURCES
**Why?**  
To read or fetch information from existing infrastructure (you donтАЩt create with it).

```hcl
data "aws_ami" "ubuntu" {
  most_recent = true
  owners      = ["099720109477"]

  filter {
    name   = "name"
    values = ["ubuntu/images/hvm-ssd/ubuntu-focal-*"]
  }
}

resource "aws_instance" "server" {
  ami           = data.aws_ami.ubuntu.id
  instance_type = "t2.micro"
}
```

---

### ЁЯМР 4. PROVIDERS
**Why?**  
They define **which platform** (like AWS, Azure, GCP) Terraform should interact with.

```hcl
provider "aws" {
  region = var.region
}
```

---

### ЁЯУВ 5. STATE FILE
**Why?**  
Terraform uses a **state file** (`terraform.tfstate`) to **track** what is created.

- It compares your code vs what exists.
- Helps with `plan`, `apply`, and `destroy`.
- DonтАЩt modify it manually.

---

### тШБя╕П 6. REMOTE BACKEND
**Why?**  
Store your **state file** in a shared location (e.g., S3) so teams can collaborate.

```hcl
terraform {
  backend "s3" {
    bucket = "my-terraform-state"
    key    = "dev/terraform.tfstate"
    region = "ap-south-1"
  }
}
```

---

### ЁЯФз 7. PROVISIONERS
**Why?**  
To run commands/scripts **after** a resource is created.

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0abcdef1234567890"
  instance_type = "t2.micro"

  provisioner "remote-exec" {
    inline = [
      "sudo apt update",
      "sudo apt install nginx -y"
    ]
  }

  connection {
    type        = "ssh"
    user        = "ubuntu"
    private_key = file("~/.ssh/id_rsa")
    host        = self.public_ip
  }
}
```

---

### ЁЯУж 8. MODULES
**Why?**  
Modules let you **reuse Terraform code** across different projects or environments.

**Structure:**
```
root/
тФЬтФАтФА main.tf
тФЬтФАтФА variables.tf
тФФтФАтФА modules/
    тФФтФАтФА ec2/
        тФЬтФАтФА main.tf
        тФЬтФАтФА variables.tf
        тФФтФАтФА outputs.tf
```

**Use Module:**
```hcl
module "my_ec2" {
  source        = "./modules/ec2"
  instance_type = "t2.micro"
  ami_id        = "ami-0abcdef1234567890"
}
```

---

### ЁЯзк 9. WORKSPACES
**Why?**  
To manage **multiple environments** like `dev`, `test`, `prod` using the same code.

**Commands:**
```bash
terraform workspace list
terraform workspace new dev
terraform workspace select dev
terraform apply
```

---

### ЁЯФР 10. SECRETS & VAULT
**Why?**  
To avoid hardcoding secrets like access keys, use **Vault** for secure secret management.

```hcl
provider "vault" {
  address = "http://127.0.0.1:8200"
  token   = var.vault_token
}

data "vault_generic_secret" "aws_creds" {
  path = "secret/aws"
}

provider "aws" {
  access_key = data.vault_generic_secret.aws_creds.data["access_key"]
  secret_key = data.vault_generic_secret.aws_creds.data["secret_key"]
  region     = var.region
}
```

## ЁЯФД Common Terraform Commands

| Command              | Description                           |
|----------------------|---------------------------------------|
| `terraform init`     | Initialize working directory          |
| `terraform plan`     | Show what Terraform will do           |
| `terraform apply`    | Apply the configuration               |
| `terraform destroy`  | Delete all infrastructure             |
| `terraform fmt`      | Format code properly                  |
| `terraform validate` | Check config correctness              |
