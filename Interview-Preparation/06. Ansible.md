# 🛠️ Ansible
---
**Ansible** is an open-source **configuration management**, **deployment**, and **orchestration** tool.  
It is designed to automate tasks such as installing software, configuring systems, and orchestrating more complex IT tasks like continuous deployments or zero downtime rolling updates.

---

## ✅ Use Case Example

A system administrator needs to install software on multiple servers.  
Doing it manually may lead to human error. Using Ansible, this task can be automated and executed on all servers simultaneously.

---

## 🔁 Alternatives to Ansible and its comparison

### ✅ **Chef** – Pull-Based Mechanism
- Uses a **pull-based** model.
- The **client (node)** pulls the configuration from the **Chef server** periodically.
- ⚙️ Example:
  > You (admin) put instructions on a shared Google Doc.  
  > Each employee is responsible for checking the doc every 30 minutes and doing what's written.


### 🐶 **Puppet** – Pull-Based Mechanism
- Also uses a **pull-based** model.
- The **Puppet Agent** periodically checks in with the **Puppet Master** to fetch and apply configurations.
- ⚙️ Example:
  > You write on a whiteboard (Puppet Master), and employees (agents) walk by and read it every 30 minutes.


### 🚀 **Ansible** – Push-Based Mechanism
- Uses a **push-based** model.
- The **Ansible server (control node)** pushes configurations directly to the client machines over SSH.
- ⚙️ Example:
  > You (admin) call each employee and tell them **right now** what to do.

## 🧑‍💻 About Ansible

- Developed by **Michael DeHaan**
- Later acquired by **Red Hat**
- Supports OS: **CentOS**, **RHEL**, **Oracle Linux**, **Debian**, and others
- **Agentless** (does not require any agent on the client systems)
- Can be used on **on-premises servers** and **cloud platforms**
- Uses **YAML** for configuration (via playbooks)

---

## ✅ Advantages

- **Free** and open-source
- **Lightweight**
- **Secure**
- Uses **push mechanism**
- **Agentless** (no client installation required)

---

## ❌ Disadvantages

- No built-in **GUI** (Graphical User Interface)
- Cannot perform **100% full automation** in certain complex scenarios

## 🔑 Key Terms in Ansible

| Term               | Description                                                                            |
|--------------------|----------------------------------------------------------------------------------------|
| **Ansible Server** | The central machine where Ansible is installed and from which tasks are executed       |
| **Modules**        | Commands or a set of commands executed on target machines (e.g., `apt`, `yum`, `copy`) |
| **Task**           | A single action to be performed on the client (e.g., install a package)                |
| **Role**           | A way to organize tasks, variables, files, templates, etc. in a reusable format        |
| **Facts**          | Information gathered from the client systems using the `gather_facts` module           |
| **Inventory**      | A file containing the list of client/host machines managed by Ansible                  |
| **Play**           | A mapping between a group of hosts and tasks (actions)                                 |
| **Playbook**       | YAML file containing plays and tasks to be executed                                    |
| **Handler**        | A special task that runs only when notified                                            |
| **Notifier**       | Trigger within a task that calls a handler if there’s a change                         |
| **Host**           | Any machine (client/node) being managed by Ansible                                     |

## 🧪 Ansible Interview Questions & Answers

---

## 1️⃣ What is Ansible and why is it useful?

**💬 Answer:**  
Ansible is a **configuration management and automation** tool. It is **agentless**, simple to use, and helps manage infrastructure using **YAML-based playbooks**. It's great for automating repetitive tasks.



## 2️⃣ How does Ansible work?

**💬 Answer:**  
Ansible connects to hosts over **SSH**, pulls an inventory file, and runs **modules** defined in **YAML playbooks** to apply configurations.



## 3️⃣ What is the difference between ad-hoc commands and playbooks?

**💬 Answer:**  
- 📌 Ad-hoc: Quick one-time commands (e.g., install a package).  
- 📘 Playbook: Full configuration or deployment scripts in YAML.

```bash
ansible all -m ping       # Ad-hoc command
ansible-playbook site.yml # Playbook
```



## 4️⃣ What are Ansible modules?

**💬 Answer:**  
Modules are the **building blocks** of Ansible. They perform specific tasks like managing files, installing packages, or controlling services.

Examples: `copy`, `yum`, `apt`, `service`, `file`, `user`, `shell`.



## 5️⃣ What is an inventory file?

**💬 Answer:**  
The inventory file contains the **list of target machines** that Ansible will manage. It can be static or dynamic.

```ini
[web]
192.168.1.10
```



## 6️⃣ What are roles in Ansible?

**💬 Answer:**  
Roles are a way to **structure your playbooks** into reusable components. A role includes tasks, variables, handlers, files, and templates organized under a directory.



## 7️⃣ How is Ansible different from Puppet or Chef?

**💬 Answer:**  
- Ansible is **agentless** (uses SSH)
- Uses **YAML** instead of custom DSLs
- Easier to learn and use
- No need for a central server (can use Ansible Tower optionally)



## 8️⃣ What is a handler in Ansible?

**💬 Answer:**  
Handlers are tasks that run **only when notified** by another task (e.g., restart service if config file changes).



## 9️⃣ What is idempotency in Ansible?

**💬 Answer:**  
Ansible ensures that no matter how many times you run a playbook, the result is the **same**. No repeated or duplicate changes unless something changes.



## 🔟 How do you secure sensitive data in Ansible?

**💬 Answer:**  
Use **Ansible Vault** to encrypt variables or entire files.

```bash
ansible-vault encrypt secrets.yml
```
